# 技能创建器

一个用于创建新技能并迭代改进的技能。

## 路径变量约定

本技能中所有涉及工作目录的路径，统一使用 `$WORKSPACE` 变量表示当前工作目录（CWD）。例如：
- 技能源码：`$WORKSPACE/skills/<skill-name>/`
- 工作空间：`$WORKSPACE/<skill-name>-workspace/`
- 临时文件：`$WORKSPACE/temp/`
- 上传文件：`$WORKSPACE/temp/uploads/<current_date>/<file_uid>.<ext>`

在实际使用中，`$WORKSPACE` 应替换为你的项目根目录的绝对路径。

---

## 整体流程

创建一个技能大致经过以下步骤：

- 确定你想让技能做什么，以及大致如何实现
- 编写技能草稿
- 创建几个测试用例，让加载了该技能的 Claude 运行它们
- 帮助用户同时从定性和定量两方面评估结果
  - 在后台跑测试的同时，如果还没有定量评估标准就先起草一些（如果已有则可以直接使用或按需修改）。然后向用户解释
  - 使用 `eval-viewer/generate_review.py` 脚本将结果展示给用户查看，同时展示定量指标
- 根据用户评估反馈重写技能（如果定量基准暴露了明显缺陷也要修复）
- 重复直到满意
- 扩大测试集，再次大规模测试

你在使用此技能时的任务是判断用户当前处于这个流程的哪个阶段，然后帮助他们推进。例如，用户说"我想做一个 X 技能"，你可以帮助他们细化需求、编写草稿、编写测试用例、确定评估方式、运行所有提示词、然后重复迭代。

另一方面，用户可能已经有了技能草稿，这时可以直接进入评估/迭代环节。

当然，你应该始终保持灵活——如果用户说"我不需要跑一堆评估，跟我随意聊就行"，那就配合他们。

技能完成后（顺序灵活），还可以运行描述优化器（有专门的脚本）来优化技能的触发效果。

## 与用户沟通

技能创建器的使用者可能涵盖各种技术背景。注意根据上下文线索调整沟通方式。默认情况下：

- "评估"和"基准测试"这类术语可以直接使用
- "JSON"和"断言"这类术语，需要先从用户的表述中确认他们了解这些概念后再使用
- 如有疑问，可以简要解释术语

## 创建技能

### 捕获意图

首先理解用户的意图。当前对话可能已经包含了用户想要封装的工作流（例如他们说"把这个变成一个技能"）。如果是这样，先从对话历史中提取答案——使用了哪些工具、步骤顺序、用户做了哪些修正、观察到的输入/输出格式。用户可能需要填补空白，且应在继续下一步前确认。

1. 这个技能应该让 Claude 能做什么？
2. 什么时候应该触发这个技能？（什么用户短语/上下文）
3. 期望的输出格式是什么？
4. 是否需要设置测试用例来验证技能？具有可客观验证输出的技能（文件转换、数据提取、代码生成、固定工作流步骤）适合设置测试用例。主观输出的技能（写作风格、艺术创作）通常不需要。根据技能类型建议合适的默认选项，但让用户决定。

### 调研和深入了解

主动询问边界情况、输入/输出格式、示例文件、成功标准和依赖项。在这些细节明确之前不要急着写测试用例。

检查可用的 MCP——如果对调研有用（搜索文档、查找类似技能、查阅最佳实践），可以通过子代理并行调研（如果可用），否则内联执行。准备好上下文以减轻用户的负担。

### 编写 SKILL.md

根据用户访谈，填写以下组件：

- **name**：技能标识符
- **description**：何时触发、做什么。这是主要的触发机制——同时包含技能的功能和使用场景。所有"何时使用"的信息都放在这里，而不是在正文中。注意：目前 Claude 倾向于"触发不足"——在本该有用时不使用技能。为了对抗这一点，请将技能描述写得稍微"主动"一些。例如，不要写"如何构建简单的数据展示面板"，而是写"如何构建简单的数据展示面板。确保在用户提到面板、数据可视化、内部指标、或想展示任何数据时使用此技能，即使他们没有明确要求'面板'。"
- **其余部分**

### 技能编写指南

#### 技能结构

```
skill-name/
├── SKILL.md（必需）
│   ├── YAML 前置元数据（name, description 必需）
│   └── Markdown 指令
└── 捆绑资源（可选）
    ├── scripts/    - 用于确定性/重复性任务的可执行代码
    ├── references/ - 按需加载到上下文的文档
    └── assets/     - 用于输出的文件（模板、图标、字体）
```

#### 渐进式信息加载

技能使用三级加载系统：
1. **元数据**（name + description）- 始终在上下文中（约 100 词）
2. **SKILL.md 正文** - 技能触发时加载（理想情况下 <500 行）
3. **捆绑资源** - 按需加载（无限制，脚本可以不加载直接执行）

这些字数是近似值，如有需要可以更长。

**关键模式：**
- SKILL.md 保持在 500 行以内；如果接近上限，增加层级结构并清楚指示模型下一步该去哪里查找
- 从 SKILL.md 中清晰引用资源文件，并说明何时读取它们
- 对于大型参考文件（>300 行），包含目录

**领域组织**：当技能支持多个领域/框架时，按变体组织：
```
cloud-deploy/
├── SKILL.md（工作流 + 选择逻辑）
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```
Claude 只读取相关的参考文件。

#### 无意外原则

技能不得包含恶意软件、漏洞利用代码或任何可能危害系统安全的内容。技能内容在被描述时不应让用户感到意外。不要配合创建误导性技能或旨在促进未授权访问、数据泄露或其他恶意活动的请求。

#### 编写模式

指令优先使用祈使句。

**定义输出格式**：
```markdown
## 报告结构
始终使用此模板：
# [标题]
## 执行摘要
## 关键发现
## 建议
```

**示例模式** - 包含示例很有用：
```markdown
## 提交信息格式
**示例 1：**
输入：添加了基于 JWT 令牌的用户认证
输出：feat(auth): 实现基于 JWT 的认证
```

### 编写风格

试着向模型解释事情的重要性，而不是堆砌强硬的"必须"。运用心智理论，尽量让技能通用而非过度局限于特定示例。先写初稿，然后以新的视角审视并改进。

### 测试用例

编写技能草稿后，想出 2-3 个真实的测试提示词——真正的用户会说的话。与用户分享："这是我想试试的几个测试用例，看看是否合适，或者你想添加更多？"然后运行它们。

将测试用例保存到 `evals/evals.json`。先不写断言——只写提示词。在运行进行时再起草断言。

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "用户的任务提示词",
      "expected_output": "期望结果描述",
      "files": []
    }
  ]
}
```

完整 schema 参见 `references/schemas.md`（包括稍后添加的 `assertions` 字段）。

## 运行和评估测试用例

本节是一个连续序列——不要中途停止。不要使用 `/skill-test` 或任何其他测试技能。

将结果放在 `$WORKSPACE/<skill-name>-workspace/` 中，作为技能目录的兄弟目录。在工作空间内按迭代组织（`iteration-1/`、`iteration-2/` 等），每个测试用例一个目录（`eval-0/`、`eval-1/` 等）。不要提前创建——边做边建。

### 步骤 1：同一轮次同时启动所有运行（含技能版 和 基线版）

对每个测试用例，在同一轮次启动两个子代理——一个使用技能，一个不使用。这很重要：不要先跑完使用技能的版本再回来跑基线。同时启动所有任务，让它们大约同时完成。

**使用技能版运行：**

```
执行此任务：
- 技能路径：<path-to-skill>
- 任务：<eval prompt>
- 输入文件：<eval files if any, or "none">
- 保存输出到：$WORKSPACE/<skill-name>-workspace/iteration-<N>/eval-<ID>/with_skill/outputs/
- 需要保存的输出：<用户关心的内容——例如 ".docx 文件"、"最终的 CSV">
```

**基线运行**（相同提示词，但基线取决于上下文）：
- **创建新技能**：完全不使用技能。相同提示词，无技能路径，保存到 `without_skill/outputs/`。
- **改进现有技能**：使用旧版本。编辑前先快照技能（`cp -r <skill-path> $WORKSPACE/<skill-name>-workspace/skill-snapshot/`），然后将基线子代理指向快照。保存到 `old_skill/outputs/`。

为每个测试用例编写 `eval_metadata.json`（断言暂时为空）。给每个评估一个基于测试内容的描述性名称——不要只用"eval-0"。目录也用这个名称。如果本次迭代使用了新的或修改过的评估提示词，为每个新目录创建这些文件——不要假设它们从上一次迭代继承。

```json
{
  "eval_id": 0,
  "eval_name": "描述性名称",
  "prompt": "用户的任务提示词",
  "assertions": []
}
```

### 步骤 2：运行期间起草断言

不要干等运行完成——利用这段时间起草每个测试用例的定量断言并向用户解释。如果 `evals/evals.json` 中已有断言，审查并解释它们检查什么。

好的断言是可客观验证的，且有描述性的名称——在基准查看器中应该一眼就能理解每个断言检查什么。主观技能（写作风格、设计质量）更适合定性评估——不要强行对需要人类判断的东西添加断言。

起草后更新 `eval_metadata.json` 文件和 `evals/evals.json` 中的断言。同时向用户解释他们将在查看器中看到什么——包括定性输出和定量基准。

### 步骤 3：运行完成时捕获计时数据

每个子代理任务完成时，你会收到包含 `total_tokens` 和 `duration_ms` 的通知。立即将此数据保存到运行目录的 `timing.json`：

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

这是捕获此数据的唯一机会——它通过任务通知传来，不会在其他地方持久化。收到通知后立即处理。

### 步骤 4：评分、聚合、启动查看器

所有运行完成后：

1. **评分** — 启动一个评分子代理（或内联评分），读取 `agents/grader.md` 并评估每个断言。将结果保存到每个运行目录的 `grading.json`。grading.json 的 expectations 数组必须使用 `text`、`passed` 和 `evidence` 字段（不是 `name`/`met`/`details` 或其他变体）——查看器依赖这些确切的字段名。对于可以编程检查的断言，编写并运行脚本，而不是目测——脚本更快、更可靠，且可跨迭代复用。

2. **聚合为基准** — 从 skill-creator 目录运行聚合脚本：
   ```bash
   python -m scripts.aggregate_benchmark $WORKSPACE/<skill-name>-workspace/iteration-N --skill-name <n>
   ```
   这会生成 `benchmark.json` 和 `benchmark.md`，包含每种配置的 pass_rate、时间和 token 统计，含均值 ± 标准差和差值。如果手动生成 benchmark.json，参考 `references/schemas.md` 获取查看器需要的确切 schema。将每个 with_skill 版本放在其基线版本之前。

3. **分析** — 阅读基准数据，发现聚合统计可能隐藏的模式。参见 `agents/analyzer.md`（"分析基准结果"部分）了解要关注什么——例如无论是否使用技能都总是通过的断言（无区分度）、高方差评估（可能不稳定）、时间/token 的权衡。

4. **启动查看器**，同时展示定性输出和定量数据：
   ```bash
   nohup python <skill-creator-path>/eval-viewer/generate_review.py \
     $WORKSPACE/<skill-name>-workspace/iteration-N \
     --skill-name "my-skill" \
     --benchmark $WORKSPACE/<skill-name>-workspace/iteration-N/benchmark.json \
     > /dev/null 2>&1 &
   VIEWER_PID=$!
   ```
   第 2 次及以后的迭代，还需传入 `--previous-workspace $WORKSPACE/<skill-name>-workspace/iteration-<N-1>`。

   **Cowork / 无头环境：** 如果 `webbrowser.open()` 不可用或环境没有显示器，使用 `--static <output_path>` 将独立 HTML 文件写入而不是启动服务器。

注意：请使用 generate_review.py 创建查看器；无需编写自定义 HTML。

5. **告知用户**："我已在浏览器中打开了结果。有两个标签页——'输出'让你逐个查看测试用例并留下反馈，'基准'显示定量对比。完成后回来告诉我。"

### 用户在查看器中看到什么

"输出"标签页一次显示一个测试用例：
- **提示词**：给出的任务
- **输出**：技能产出的文件，尽可能内联渲染
- **上一次输出**（第 2 次迭代起）：折叠区域显示上次迭代的输出
- **正式评分**（如果运行了评分）：折叠区域显示断言通过/失败
- **反馈**：自动保存的文本框
- **上一次反馈**（第 2 次迭代起）：上次的评论，显示在文本框下方

"基准"标签页显示统计摘要：每种配置的通过率、计时和 token 使用量，含每个评估的分解和分析师观察。

导航使用上一个/下一个按钮或方向键。完成后点击"提交所有评审"，将所有反馈保存到 `feedback.json`。

### 步骤 5：读取反馈

用户告诉你他们完成了，读取 `feedback.json`：

```json
{
  "reviews": [
    {"run_id": "eval-0-with_skill", "feedback": "图表缺少坐标轴标签", "timestamp": "..."},
    {"run_id": "eval-1-with_skill", "feedback": "", "timestamp": "..."},
    {"run_id": "eval-2-with_skill", "feedback": "完美，非常好", "timestamp": "..."}
  ],
  "status": "complete"
}
```

空反馈意味着用户认为没问题。把改进重点放在用户有具体意见的测试用例上。

完成后关闭查看器服务器：

```bash
kill $VIEWER_PID 2>/dev/null
```

## 改进技能

这是循环的核心。你已运行了测试用例，用户已审查结果，现在需要根据反馈改进技能。

### 如何思考改进

1. **从反馈中泛化。** 这里的大局是：我们试图创建可以被使用上百万次的技能。你和用户只在少数几个示例上反复迭代，因为这样更快。用户非常熟悉这些示例，可以快速评估新输出。但如果技能只对这些示例有效，就没用了。与其加入琐碎的过拟合修改或过度约束的"必须"规则，如果有顽固的问题，不如尝试分支出来使用不同的隐喻或推荐不同的工作模式。尝试的成本相对较低，也许能找到很棒的方案。

2. **保持提示词精简。** 删除没有实质贡献的内容。确保阅读执行记录，而不仅仅是最终输出——如果看起来技能让模型浪费大量时间做无用功，可以尝试删除导致这种情况的技能部分。

3. **解释原因。** 努力解释你要求模型做的每件事背后的**原因**。当今的 LLM 很聪明，有良好的心智理论，在好的框架下能超越机械执行指令。即使用户的反馈简短或有些沮丧，也要真正理解任务和用户写的内容的原因，然后将这种理解传达到指令中。如果你发现自己在写全大写的"始终"或"绝不"，或使用超级死板的结构，这是一个警告信号——如果可能，重新表述并解释推理，让模型理解为什么你要求的东西很重要。

4. **寻找测试用例间的重复工作。** 阅读测试运行的执行记录，注意子代理是否都独立编写了相似的辅助脚本或采取了相同的多步方法。如果 3 个测试用例都导致子代理编写 `create_docx.py` 或 `build_chart.py`，这是强烈的信号——技能应该捆绑该脚本。写一次，放在 `scripts/` 中，告诉技能使用它。

### 迭代循环

改进技能后：

1. 将改进应用到技能
2. 将所有测试用例重新运行到新的 `iteration-<N+1>/` 目录，包括基线运行
3. 启动带有 `--previous-workspace` 指向上一次迭代的查看器
4. 等待用户审查完毕
5. 读取新反馈，再次改进，重复

继续直到：
- 用户表示满意
- 反馈全为空（一切看起来都好）
- 没有实质性进展

## 进阶：盲测对比

在需要更严格比较两个技能版本时（例如用户问"新版本真的更好吗？"），有一个盲测对比系统。阅读 `agents/comparator.md` 和 `agents/analyzer.md` 了解详情。基本思路是：将两个输出交给一个独立代理，不告诉它哪个是哪个，让它判断质量，然后分析赢家为何胜出。

这是可选的，需要子代理，大多数用户不需要。人工审查循环通常就够了。

## 描述优化

SKILL.md 前置元数据中的 description 字段是决定 Claude 是否调用技能的主要机制。创建或改进技能后，可以提议优化描述以提高触发准确率。

### 步骤 1：生成触发评估查询

创建 20 个评估查询——应触发和不应触发的混合。保存为 JSON：

```json
[
  {"query": "用户提示词", "should_trigger": true},
  {"query": "另一个提示词", "should_trigger": false}
]
```

查询必须真实，是 Claude Code 或 Claude.ai 用户真正会输入的内容。不是抽象请求，而是具体的、有细节的请求——文件路径、关于用户工作或情况的个人背景、列名和值、公司名称、URL。一些可能是小写的、包含缩写或打字错误或口语化表达。使用不同长度的混合，重点放在边界情况而非明显的案例上。

**反面示例：**`"格式化这个数据"`、`"从 PDF 中提取文本"`、`"创建一个图表"`

**正面示例：**`"我老板刚发给我一个 xlsx 文件（在我下载目录里，好像叫 'Q4 sales final FINAL v2.xlsx'），她要我加一列显示利润率百分比。收入在 C 列，成本好像在 D 列"`

对于**应触发**查询（8-10 个），考虑覆盖度。你需要同一意图的不同措辞——有些正式，有些随意。包含用户没有明确提到技能或文件类型但明显需要它的案例。

对于**不应触发**查询（8-10 个），最有价值的是"差点就该触发的"——与技能共享关键词或概念但实际需要不同东西的查询。不要让不应触发的查询明显无关——"写一个斐波那契函数"作为 PDF 技能的负面测试太简单了。负面案例应该确实很有迷惑性。

### 步骤 2：与用户审核

使用 HTML 模板将评估集呈现给用户审核：

1. 读取 `assets/eval_review.html` 模板
2. 替换占位符：
   - `__EVAL_DATA_PLACEHOLDER__` → 评估项的 JSON 数组
   - `__SKILL_NAME_PLACEHOLDER__` → 技能名称
   - `__SKILL_DESCRIPTION_PLACEHOLDER__` → 技能当前描述
3. 写入临时文件（如 `$WORKSPACE/temp/eval_review_<skill-name>.html`）并打开
4. 用户可以编辑查询、切换 should-trigger、增删条目，然后点击"导出评估集"
5. 文件下载为 `eval_set.json`

### 步骤 3：运行优化循环

告诉用户："这需要一些时间——我会在后台运行优化循环并定期检查进度。"

将评估集保存到工作空间，然后在后台运行：

```bash
python -m scripts.run_loop \
  --eval-set <path-to-trigger-eval.json> \
  --skill-path <path-to-skill> \
  --model <model-id-powering-this-session> \
  --max-iterations 5 \
  --verbose
```

使用当前会话的模型 ID，以确保触发测试匹配用户实际体验。

运行期间定期查看输出，向用户更新当前在哪个迭代以及分数情况。

此脚本自动处理完整的优化循环。它将评估集分为 60% 训练集和 40% 保留测试集，评估当前描述（每个查询运行 3 次以获得可靠的触发率），然后调用 Claude 根据失败案例提出改进建议。它在训练集和测试集上重新评估每个新描述，最多迭代 5 次。完成后，它会打开一个 HTML 报告显示每次迭代的结果，并返回包含 `best_description` 的 JSON——按测试集分数而非训练集分数选择，以避免过拟合。

### 技能触发的工作原理

理解触发机制有助于设计更好的评估查询。技能以 name + description 出现在 Claude 的 `available_skills` 列表中，Claude 根据该描述决定是否查阅技能。关键要知道的是，Claude 只对自己不能轻松独立处理的任务查阅技能——简单的一步查询如"读取这个 PDF"可能不会触发技能（即使描述完美匹配），因为 Claude 可以用基础工具直接处理。复杂的、多步骤的或专业化的查询在描述匹配时会可靠地触发技能。

这意味着你的评估查询应该足够复杂，让 Claude 确实会从查阅技能中受益。

### 步骤 4：应用结果

从 JSON 输出中取出 `best_description` 并更新技能 SKILL.md 的前置元数据。向用户展示前后对比并报告分数。

检查是否有 `present_files` 工具。如果没有，跳过此步骤。如果有，打包技能并向用户展示 .skill 文件：

## Claude.ai 特定说明

在 Claude.ai 中，核心工作流相同（起草 → 测试 → 审查 → 改进 → 重复），但由于没有子代理，一些机制有变化：

**运行测试用例**：没有子代理意味着无法并行执行。对每个测试用例，读取技能的 SKILL.md，然后按其指令完成测试提示词。逐一执行。跳过基线运行——直接使用技能完成任务。

**审查结果**：如果无法打开浏览器，跳过浏览器查看器。在对话中直接展示结果。对每个测试用例，显示提示词和输出。如果输出是用户需要查看的文件（如 .docx 或 .xlsx），保存到文件系统并告知位置。内联请求反馈。

**基准测试**：跳过定量基准——它依赖基线比较，在没有子代理时无意义。聚焦用户的定性反馈。

**迭代循环**：与之前相同——改进技能、重新运行测试用例、请求反馈——只是中间没有浏览器查看器。

**描述优化**：此部分需要 `claude` CLI 工具（具体是 `claude -p`），仅在 Claude Code 中可用。如果在 Claude.ai 上则跳过。

**盲测对比**：需要子代理。跳过。

**打包**：`package_skill.py` 脚本只需要 Python 和文件系统就能工作。

**更新现有技能**：用户可能要求更新现有技能。在这种情况下：
- **保留原始名称。** 注意技能的目录名和 `name` 前置元数据字段——原样使用。
- **编辑前复制到可写位置。** 已安装的技能路径可能是只读的。复制到 `$WORKSPACE/temp/skill-name/`，在那里编辑，然后打包。

## Cowork 特定说明

如果你在 Cowork 中：

- 你有子代理，所以主工作流（并行启动测试用例、运行基线、评分等）都可以工作
- 没有浏览器或显示器，生成查看器时使用 `--static <output_path>` 写入独立 HTML 文件
- 生成查看器后反馈通过下载 `feedback.json` 文件收集
- 打包可以正常工作
- 描述优化应该可以正常工作（使用 `claude -p`）
- **更新现有技能**：遵循 Claude.ai 部分的更新指南
