第 1 部分

1. 概述

Priva 最初是一个基于 Claude Agent SDK 的单机个人助手单体应用。Priva Cloud 是对它的一次 有意「分支」（fork）——把单体改造成面向数百名半可信内部用户的 Kubernetes 多租户平台：每个用户一个可缩容至零的 Agent Pod，统一藏在一个无状态前门之后。

整套设计都围绕一个来之不易的「分离」，而代码库里到处都在重复它：

一次 Agent 交互的 字节 沿 browser ═ agentgateway ═ agent-runner pod 流动，从不经过控制面。 决策——你是谁、用哪个 Pod、唤醒它、给你令牌——则由边缘每请求咨询一次的 Agent-runtime EndPoint Picker（EPP） 在带外完成。

这次迁移是教科书式的 绞杀者模式（strangler-fig）：一个约 6.9K 行的单体（priva/api， 3 个 OS 进程）被逐步切分为多个可独立部署、独立伸缩的服务，彼此只通过 gRPC / HTTP / Redis 通信——绝不进程内 import。截至当前 main，4 个后端服务 + 一个共享契约库 + 一个启动器已构建并跑在 minikube 上， 并已端到端验证过一次完整的 Agent 交互穿过真实边缘。两个服务（channel-connector、scheduler）和一个 （state-reader）仍在路线图上。

维度说明
目标把单机 Priva fork 成 K8s 多租户；开源版 Priva 保持不动（决策 14）。
租户模型每用户一个 Pod + PVC + RBAC + NetworkPolicy；空闲时 缩容至零（决策 2）。
前门面向所有入口的无状态边缘（agentgateway）；不持有任何 Agent 状态。
核心单元会话（session）——用户卷上的一个 JSONL 转录文件，由不可变的 session_uuid 标识。
身份一个中心账户；浏览器 / WeCom / Feishu 是关联身份（每用户自带 LLM 密钥，BYOK）。
方法论26 条锁定决策 → 6 份组件深潜 → 逐文件切分图 → 7 个渐进阶段。
第 2 部分

2. 核心设计思想

两个想法撑起了整个架构。理解了它们，其余部分都读作推论。

2.1 · Byte path ≠ 控制路径

天真的网关会把一切都代理穿过自己，于是变成有状态瓶颈。Priva 的边缘改为 转发：对每个运行时请求， 它向 EPP 做一次旁路调用，询问 「用哪个 Pod，并唤醒它」，拿回目标地址 + 一个签名令牌， 然后把请求直接流式发往那个 Pod。边缘 永远不在Byte path的远端——它传字节但不决策；EPP 决策但不碰字节。

2.2 · 唯一不可变的 session_uuid，绝不用 CLI 派生的 id

相较天真 fork 的那个最关键修正：每个 key、锁、路由、绑定都以 平台生成的稳定 session_uuid 为键。曾以为 claude CLI 会在运行途中改写自己的 session id——那将让所有锁和路由变成孤儿。修法 （决策 M2）：session_uuid 就是 CLI 的 --session-id，在会话生命周期内不可变—— CREATE 传 --session-id<uuid>，RESUME 传 --resume<uuid>，两者绝不混用。 无 remap，无可变的 cli_session_id 列。

为什么重要
这两个不变量把原单体的脑裂（各自独立的 channel-daemon 与 API-bridge 注册表 + 协调器）化解为 一个 路由层，其唯一持久状态在 SQLite + Redis；并让每个协调结构都能用单一稳定 key 寻址。

有意延后（且可逆）

M6 — BYOK + 不计量

每个用户自带 真实的 LLM 密钥；没有共享组织密钥、没有虚拟密钥代理、 没有预算账本。计量代理组件被 移除——密钥变成一个由 operator 注入的普通 Secret。token 用量由 Pod 自报（仅用于观测）。

出口网关延后

Pod 直连 LLM / MCP / 网络——无白名单、无强制代理穿透。Pod 到 Pod 的默认拒绝仍然保留。可逆：日后重新加回正向代理即可恢复受控出口与计量。

EPP 如何定位到对应的 Pod —— 端点发现的 Byte path

承上：上面第 ②–⑥ 步「解析 → 唤醒 → 引导」里最关键的问题是——EPP 究竟 怎么 知道账户 X 的请求 该发往哪个 agent-runner Pod？答案不靠注册表、也不靠每账户 DNS，而靠 确定性身份 加一个写在 CR 上的 会合点。下面单独把这条「端点发现」路径画出来。

端点发现 —— 逐步

关键：靠「确定性身份 + 会合点」定位，而非注册表
EPP 不查服务注册表、也不做每账户 DNS 解析。Pod 的定位只靠两点：(1) 确定性身份——account_id 同时是 AgentTenant CR 名、对象名 ar-<account>、以及 Pod 标签 priva.io/account-id； (2) 会合点——operator 把活的 podIP 写进 CR 的 status.podIP，EPP 只读它。真正按 标签查询 K8s API 的是 operator（wait_pod_ready），EPP 拿到的是已解析好的结果。InferencePool 的 app=agent-runner 选择器只圈定 候选集；最终唯一终点完全由 EPP 返回的 destination 头 决定（FailClose：EPP 不给端点，请求即失败，绝不乱投）。返回的是 Pod IP 而非 Service ClusterIP ——一次直达 Pod 的跳转。

Agent-runtime EndPoint Picker（EPP）——整个决策浓缩成一个函数

EPP 就是 GIE InferencePool 的端点选择器（EndPoint Picker），经 gRPC 以 Envoy 外部处理（ext_proc）形态提供。引人注目的是，它的核心决策只有约 25 行—— control-panel/extproc.py 里的 handle_request_headers——恰好就是设计所要的 resolve · mint · wake · steer：

async def handle_request_headers(http_headers):
  token = bearer or subprotocol("priva.token.") or ?token=  # 3 个来源
  user = await authenticate_raw_token(token)          # 解析
  if user is None: return _immediate(401)
  endpoint = await to_thread(provisioner.wake_and_wait, account_id) # 唤醒
  if not endpoint: return _immediate(503, "waking, retry")
  return _steer(endpoint, mint(account_id, username))     # 签发 + 引导

四个关键坑——让路径真正跑通的东西

要把字节穿过一个第三方 Rust 边缘、送达一个从零伸缩的 Pod，暴露出四个非显然的互通 bug。每个都作为实际构建的 「坑」被记录，并在代码中修复：

1 · EPP 必须用 TLS
agentgateway 经 TLS 拨号 InferencePool 的 EndpointPicker（GIE 约定，集群内 skip-verify）。 明文 EPP 会以 InvalidContentType 失败。修法：自签名 h2 上下文（ALPN h2），用 grpclib（纯 Python HTTP/2）提供——因为 C-core 的 grpc.aio 服务端与该 Rust 客户端不互通。 extproc.py:173
2 · 把 body 原样回传
GIE ext_proc 路径会为 EPP 缓冲请求 body。一个空的 BodyResponse 会 丢掉 body → Pod 收到空 body → 422。修法：尽量用 mode_override 跳过 body；仍有分片到来时，用 body_mutation 把字节原样回传。extproc.py:113
3 · 在升级请求上做 WS 鉴权
WebSocket 升级请求没有 body、也没有 Authorization 头。JWT 改为随 Sec-WebSocket-Protocol 列表作为 priva.token.<jwt> 携带；runner 在握手时只回显 priva.ws.v1——令牌不进 URL、不进网关访问日志。extproc.py:78 · deps.py
4 · IS_SANDBOX=1
运行时以 --dangerously-skip-permissions 驱动 CLI，而 CLI 在 root 下会拒绝该参数。每账户 Pod 本身就是沙箱，于是 operator 在 Pod 环境里设 IS_SANDBOX=1——这是 CLI 认可的逃生口—— 否则每次运行都会以 1 退出。operator/kube.py

信任链

Pod 只信任 EPP、且仅信任 EPP，靠一个短 TTL 的签名令牌。这是开发模式的接缝；mint/verify 这套接口设计成可在生产换成 JWKS / mTLS 而不动调用方。

步骤机制
签发（EPP）HS256 JWT {account_id, username, exp:+60s}，用 dataspine.api_key_hmac_secret（回退 auth.jwt_secret）签名。runner_token.py:31
携带在 EPP 引导响应里设为 x-priva-runner-token 头；随请求一同到达 Pod。
校验（Pod）runner 解码并校验签名/过期；要求带 account_id + username 声明。signed_header.py → runner_token.verify
绑定声明里的 account_id 必须与 Pod 固定的 ACCOUNT_ID 环境变量一致——一个 Pod 永远只能为它那一个租户行动。deps.py:42
第 6 部分

6. 状态存储在哪

三个存储，三种持久性契约。来自设计文档的心智模型： data-spine = 由图书管理员看守的唯一账本 · Redis = 人人直接张贴的公告板 · PVC = 每个 Pod 的私人抽屉。

① data-spine — SQLite

持久、共享、单写入。只通过数据面客户端访问（当前进程内 → 集群内 gRPC）。 WAL、foreign_keys=ON、STRICT、RWO PVC、Recreate。

② Redis — 协调

网络化、多客户端、临时/AOF 分层。锁、权限索引、路由、在线状态。由每个服务 直接 访问；key 名称经 redis_catalog 共享。第4阶段

③ PVC — Pod 自有文件

每 Pod RWO。会话 JSONL 转录、CLI 原生读取的运行时/MCP/hook 配置、临时上传、审计 JSONL。只在 Pod 存活时编辑。

关系核心——account 是枢纽

最初的 16 表设计被有意收敛到 5 张锁定的第 1 阶段表；随后在 minikube 切片中又加回一张 secret 表以承载 BYOK 凭据供 operator 注入（实际为 6 张）。一切都挂在 account_id 上、 ON DELETE CASCADE——一个干净的爆炸半径。

表基数持有备注
account ★枢纽username、password（bcrypt）、api_key（Fernet）、role、status、feishu 关联api_key_lookup = HMAC，免解密即可 O(1) 查找
quota1 : 1tier、max_concurrent_sessions、idle_grace运营杠杆
channel_binding1 : 0..1session_uuid（UNIQUE）、first_run_done全新；session_uuid ⇄ 会话的纽带
scheduled_job1 : Ntrigger（JSON）、job_type、job_configcron / interval / 一次性
job_run_record1 : Nstatus、duration、num_turns、result_summary仅元数据——转录留在 PV；删除任务时 job_id SET NULL
secret 切片1 : 1bundle（Fernet 密文）、generation读取时解密；唤醒时物化为一个 K8s Secret
为什么不把转录放进数据库？
会话 JSONL 是追加密集、按用户、且 CLI/SDK 原生读取这些文件——数据库副本会冗余且可能漂移。Postgres 持有 跨用户查询的事实；PV 持有 Agent 读写的字节。转录内容绝不复制进关系存储（决策 M5：文件系统 就是会话存储；没有 session_index 表）。
第 7 部分

7. 代码映射 —— 从单体到 Monorepo

切分是可逆的绞杀者。每个现有应用文件都被赋予一个目标可部署单元和一个改造动词；单体会一直可启动， 直到第 4 阶段 connector/scheduler 抽离。

反转——一切赖以悬挂的接缝

在单体里，进程内的运行循环 agent_run_events（claude_sdk/service.py）有 三个进程内调用方：WebUI agent 路由、scheduler 守护进程、channels 守护进程。切分 砍掉其中两个：

现在（耦合）

agent router   ─┐
scheduler daemon ─┼─ 进程内 ─▶ run
channels daemon  ─┘

之后（派发器）

agent router（Pod 内）─┐
scheduler ─ 派发 ──────┼─▶ Pod 跑这次交互
connector ─ 派发 ──────┘

切分后，只有 agent-runner 镜像装载 service.py。scheduler 向 inbox RPUSH 并 CR-patch 唤醒；connector 调用 EPP 后直拨被唤醒的 Pod。第 4 阶段的验收测试就是字面意义上的：两个服务都不再 import claude_sdk。这也是为什么 scheduler 的执行器被 重定位 到 Pod——它们同样在跑 Agent。

改造动词（如何读这张图）

LIFT（搬运）

文件原样移动（如 models/*、hooks 执行器）。

RESHAPE（重塑）

移动 + 改行为：存储→RPC、进程内→派发、单例→每 Pod。

SPLIT（拆分）

一个路由的多个路由分到不同单元（双面路由）。

RELOCATE（重定位）

搬到非显然的归宿（scheduler 执行器 → Pod）。

DELETE（删除）

由平台机制替代（server.sh、config_store）。

第 9 部分

9. 关键洞察

除功能清单之外，这套代码库的可圈可点之处。

决策先行的工程

代码库罕见地以文档为引领：26 条锁定决策 → 6 份组件深潜 → 一张逐文件切分图 → 一次在写码 之前 就 预演脑裂、僵尸 Pod、双权威锁的「对抗性设计风险」复盘。文档还明确标出后续深潜在哪里 取代 了蓝图 （M1/M2/M5/M6），让计划对自身的修订保持诚实。

边缘转发、EPP 决策

把Byte path与控制路径分离，是杠杆最大的一个选择。它让前门保持无状态（N 副本，无 Future、无 JSONL），让 缩容至零变成每请求一次的唤醒，也意味着对延迟敏感的 EPP 日后可以剥离成独立单元，而不触碰任何承载字节的东西。

一个不可变 id 消灭一整类 bug

把每个锁、路由、绑定、审批都以平台生成、且 就是 CLI --session-id 的 session_uuid 为键——配合 CREATE/RESUME 拆分、无 remap——从根上移除了原本的孤儿 id 竞态， 而非修补其症状。

按持久性契约安置状态

三个存储，各取所长：SQLite（单一所有者的账本）、Redis（网络化原子协调）、PVC（CLI 原生读取的文件）。 「绝不把转录字节复制进关系存储」这条规则避免了漂移，也让数据库保持为跨用户事实的轻量索引。

对 alpha 状态足够坦诚

实际构建的传输除那一跳被强制的 TLS ext_proc 外处处明文；HS256 共享密钥令牌暂代 JWKS/mTLS；BYOK + 计量 延后。关键在于：每个权宜之计都坐在一条稳定接缝之后（mint/verify、wake/steer 接口），所以生产化替换是局部的，而非重写。

可运维性内建

确定性的 K8s 命名（ar-<account>）使得无需注册表；operator 是 唯一 伸缩者，于是只有一条 唤醒通道；/health 暴露 active_runs + last_activity_ts 以支撑简单的空闲回收； 每个服务都暴露 /metrics；priva-cloud 启动器经入口点发现，统一派发到所有服务。

风险与关注项

  • 单体仍承载 IM + 定时。第 4 阶段落地前，channels 与 scheduler 守护进程仍住在休眠的 priva/api 树里——平台尚未完全自托管这些入口。
  • Redis 分层已设计，但尚未接线。协调底座（inbox、审批索引、租约）随 connector/scheduler 到来； 跨副本的权限会合依赖它。
  • data-spine 单写入者。replicas:1 + Recreate 对 SQLite 是正确的，但也是 吞吐/可用性上限；PgRepo 接口已为将来的 Postgres 路径打桩。
  • alpha 安全态势。集群内明文 gRPC、自签名 skip-verify TLS、共享密钥 runner 令牌、放松的出口—— 都是刻意为之且接缝隔离，但在面向不可信暴露之前都必须翻转。
